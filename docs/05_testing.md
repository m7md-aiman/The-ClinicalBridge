# Module 5 — Testing & Evaluation Framework

*Part of the ClinicalBridge prompt-engineering portfolio. Maps to **M5: Testing LLM Applications**.
Implemented in [`src/clinicalbridge/evaluation/`](../src/clinicalbridge/evaluation/),
[`scripts/run_eval.py`](../scripts/run_eval.py), and the `tests/` suite.*

---

## 1. Two layers of testing

**Unit/integration tests (pytest).** Every component has tests; agent tests have an offline part
(prompt/format/guardrail logic) and a live part (real OpenRouter calls), the latter auto-skipped
when no API key is present. Deterministic pieces (schemas, tools, memory, RAG chunking, metrics) are
fully offline. Run: `pytest -q`.

**Scenario evaluation (the dashboard).** The five gold scenarios are run end-to-end through the
orchestrator and scored against their rubrics. Run: `python scripts/run_eval.py`.

## 2. Metrics (defined *before* building — see the gold rubrics in Phase 5)

Each scenario carries an evaluation rubric (`acceptable_urgencies`, `must_include`, `must_flag`,
`must_avoid`). The scorer ([`evaluation/metrics.py`](../src/clinicalbridge/evaluation/metrics.py))
produces:

| Metric | What it measures | Why it matters |
|---|---|---|
| **urgency_ok** | final urgency ∈ acceptable set | clinical triage correctness (and not over/under-calling) |
| **must_include_coverage** | fraction of required facts present | did synthesis surface the key context? |
| **must_flag_coverage** | required gaps/uncertainties flagged | honest handling of missing data |
| **must_avoid_violations** | forbidden phrasing present | e.g. never accuse the patient (PT-005) |
| **citation_coverage** | analytical statements that carry ≥1 source | anti-hallucination discipline |
| **hallucination_count** | cited sources NOT in the allowed set | the core safety metric — should be **0** |
| **latency_seconds** | wall-clock per scenario | clinical usefulness (<60 s goal) |

A scenario **passes** when: urgency is acceptable, zero hallucinated citations, zero must-avoid
violations, must-include coverage ≥ 0.6, and citation coverage ≥ 0.9.

## 3. The matching lesson (Phase 5 bug, fixed here)

A subtle but important evaluation pitfall: naive substring matching reported the forbidden word
"lying" inside "imp**lying**". The framework therefore uses **two matchers**:
- `must_include` / `must_flag` → **lenient substring** (find the concept; "diet" matches "dietary").
- `must_avoid` → **strict word-boundary** (`(?<!\w)term(?!\w)`) so "lying" never matches "implying".

This asymmetry is unit-tested ([`tests/test_evaluation.py`](../tests/test_evaluation.py)) and the
gold briefs are scored against their own rubrics as a sanity floor.

## 4. Known-correct test cases

The five scenarios are the regression set, each with a hand-authored gold brief and a documented
"lesson" (missed medication, false alarm, silent deterioration, incomplete record, conflicting
data). Because the dataset is **deterministic** (fixed seed), data-quality and prompt-quality
failures stay distinguishable across runs.

## 5. Results

Generated by `python scripts/run_eval.py` (writes `eval/results/evaluation.json` and prints a
dashboard). The latest run is summarized in
[`docs/evaluation_results.md`](evaluation_results.md), including the headline safety result
(hallucinated citations) and any per-scenario gaps that feed the Module-3 prompt iteration
(Phase 15).

> The evaluation is the bridge from "it runs" to "it works, measurably" — and, just as importantly,
> it pinpoints *where* it doesn't yet, which is the input to the next round of prompt engineering.
